## 🟠 HIGH (Should Fix)

### Issue 1: Unbounded Vector Pre-allocation
- **File:** `src/lib.rs:205` and `src/lib.rs:252`
- **Problem:** `Vec::with_capacity(size)` uses a size value directly from the network packet without validation. A malicious or malformed packet could specify a huge size, causing an OOM panic.
- **Fix:** Add a sanity check for `size` before allocation (e.g., max 10MB or reasonable buffer limit).

### Issue 2: Potential Infinite Loop in `receive_chunk`
- **File:** `src/lib.rs:214`
- **Problem:** If the `transport` read returns 0 bytes (EOF) without error, the loop condition `remaining > 0` might persist if logic isn't handling explicit disconnects, potentially spinning. `read_exact` handles EOF by returning an error, which is good, but `UdpSocket::recv` might return 0.
- **Fix:** Ensure explicit handling for 0-byte reads if `UdpSocket` is used in a way that permits empty datagrams (though less critical for `read_exact` on TCP).

## 🟡 MEDIUM (Recommended)

### Issue 1: Magic Number Usage
- **File:** `src/lib.rs:271`
- **Problem:** Hardcoded `0xFFC0` and `16 * 1024`.
- **Fix:** Move these to `src/constants.rs` as named constants like `TCP_MAX_CHUNK` and `UDP_MAX_CHUNK`.

### Issue 2: Error Handling in `connect`
- **File:** `src/lib.rs:88`
- **Problem:** The `connect` method doesn't handle the `CMD_ACK_UNAUTH` case fully (commented out).
- **Fix:** Implement the auth logic or at least return a specific error rather than just logging/ignoring.

### Issue 3: String Decoding Strategy
- **File:** `src/lib.rs:374`
- **Problem:** `String::from_utf8_lossy` allocates a new string. For high-throughput parsing, this might be slow.
- **Suggestion:** Consider returning `Cow<str>` or using `str::from_utf8` if valid UTF-8 is guaranteed/expected.
